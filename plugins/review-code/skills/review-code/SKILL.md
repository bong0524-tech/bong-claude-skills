## 목적
커밋·머지 직전, 이번에 **새로 들어간 코드**가 깨끗한지 한 번에 점검한다.
스타일·위생·컨벤션부터 보안·로직·Git 위생까지 자체 완결로 본다 (다른 스킬에 위임하지 않음).

## 범위 (중요)
- **미커밋 변경만** 본다. 이미 커밋된 코드, diff 밖 인접 코드는 건드리지 않는다.
- 변경 파일: `git diff HEAD` 의 **추가된 줄(`^+`)** 만 점검 대상.
- 신규(untracked) 파일: 파일 **전체**를 점검.
- 근거는 항상 diff에 있어야 한다. **추측으로 이슈를 지어내지 않는다.**

## 절차
1. 범위 파악:
   ```
   git status --short
   git diff HEAD              # 추적 파일 변경
   ```
   신규 파일은 `git status`로 식별해 내용을 Read.
2. 아래 **점검 항목**을 카테고리별로 돌며, 발견 시 `파일경로:라인` 으로 지목한다.
   diff 추가 줄만 빠르게 훑는 예: `git diff HEAD -U0 | grep -nE '^\+...'`
3. 빌드 검증: 프로젝트 빌드툴을 감지해 **컴파일/타입체크**를 실행한다 (아래 표).
   출력의 **마지막 5~10줄을 코드블록으로 그대로 인용**한다. 인용 없이 "통과" 주장 금지.
4. Git 위생 점검 (커밋 메시지·추적 파일·커밋 크기).
5. 카테고리별로 **발견 / 미발견**을 명시하고, 이슈는 **🔴 blocker / 🟡 권장**으로 우선순위를 매겨 보고한다. 마지막에 **커밋 가능 여부**를 한 줄로 판정한다.

## 점검 항목

### A. 네이밍 · 스타일
- **함수/변수 네이밍**: 프로젝트 기존 패턴(camel/snake/PascalCase)과 일치하는가. 의미 없는 이름(`tmp`, `data2`, `aaa`) 금지.
- **약어 일관성**: 한 코드베이스에서 `cfg`/`config`, `msg`/`message` 혼용 금지. 기존 표기를 따른다.
- **불리언 네이밍**: `is/has/can/should` 접두 + **긍정형**(`isEnabled`, not `isNotDisabled`).
- **주석 스타일**: WHAT이 아니라 WHY를 설명. 형식이 주변과 일관. 코드를 주석으로 막아둔 흔적 금지.
- **파일/폴더 네이밍**: 디렉토리·파일명이 기존 규칙·계층과 일치.

### B. 잔여물 · 군더더기
- **디버그 잔여물**: `System.out.print*`·`printStackTrace`·`console.log/debug`·`print(`·`debugger`·`dump`·`var_dump`.
- **머지 충돌 마커**: `<<<<<<<`, `=======`, `>>>>>>>`.
- **임시/테스트용 코드**: 강제 `true/return`, 인증·검증 우회, 하드코딩된 테스트 값, 실험용 주석 블록.
- **죽은 코드**: 이번 변경이 만든 미사용 import·변수·함수, 도달 불가 분기.
- **TODO/FIXME/XXX/HACK** 잔존.

### C. 하드코딩 · 보안
- **하드코딩**: 매직 넘버/문자열, 환경별 URL·경로·포트·IP. 상수/설정으로 빼야 할 것.
- **시크릿**: API 키·비밀번호·토큰·private key 가 코드에 직접 박혔는지.
- **민감정보 로깅**: 비밀번호·개인정보·토큰이 로그로 새는지.

### D. 로직 · 견고성
- **분기 누락**·else 미처리, **null/빈값/경계값** 미처리.
- **예외 삼킴**: 빈 `catch {}`, 근본 원인을 숨기는 catch-all.
- **입력 검증** 누락, 동시성·인코딩(UTF-8) 이슈.

### E. 빌드·동작 보장 (감지 후 실행)
| 감지 파일 | 컴파일/타입체크 명령 |
|---|---|
| `gradlew` | `./gradlew compileJava` |
| `pom.xml` | `mvn -q compile` |
| `package.json`(+tsconfig) | `npx tsc --noEmit` 또는 `npm run build`/`lint` |
| `Cargo.toml` | `cargo check` |
| `go.mod` | `go build ./...` |
| `Makefile` | `make`(build/check 타깃) |

여러 개면 사용자에게 어느 것인지 묻는다. 임의로 정하지 않는다.

### F. Git 위생
- **커밋 메시지 컨벤션**: 프로젝트 규칙을 `git log --oneline -10`으로 확인 후 동일 형식 권고 (예: `type(scope): 요약`).
- **불필요한 파일 추적**: 빌드 산출물·IDE 설정·`.env`/시크릿·로그·대용량·`.DS_Store` 가 스테이징됐는지. `.gitignore` 누락 점검.
- **커밋 단위 크기**: 무관한 변경이 한 커밋에 섞였으면 분리 권고.

## 보고 형식
```

### E. 빌드 — ✅ 통과
(컴파일 출력 마지막 7줄 인용)

...

### 판정: 🔴 커밋 보류 (blocker 1건 해결 필요) / ✅ 커밋 가능
```

## 금지
- diff 밖 코드 지적, 요청 범위 밖 리팩터링 제안.
- diff에 근거 없는 추측성 이슈.
- 빌드 출력 인용 없이 "통과" 주장.
- 이슈를 직접 수정하지 않는다. **점검·보고만** 한다 (수정은 사용자 지시 후).
